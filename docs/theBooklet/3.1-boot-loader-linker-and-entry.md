# Chapter 3.1 — Boot Loader, Linker, and Kernel Entry

This chapter connects your boot files into one clear execution story.

## Source Files

- `loader.s`
- `linker.ld`
- `makefile`
- `kernel.cpp` (`callConstructors`, `kernelMain`)

## What Happens First

### 1) GRUB loads your ELF kernel

Your multiboot header in `loader.s` (`MAGIC`, `FLAGS`, `CHECKSUM`) makes the kernel discoverable by GRUB.

### 2) Entry symbol

`linker.ld` sets `ENTRY(loader)`, so execution begins at `loader:` label.

### 3) Early assembly setup

`loader.s`:

- moves stack pointer to `kernel_stack`
- calls `callConstructors`
- pushes multiboot registers and calls `kernelMain`

### 4) C++ global constructors run

`callConstructors` iterates from `start_ctors` to `end_ctors` (defined via linker script `.init_array`).

## Linker Layout (Why It Matters)

`linker.ld` places kernel at `0x0100000` and defines sections:

- `.text` includes multiboot header and code
- `.data` includes ctor table
- `.bss` holds uninitialized storage

This ensures deterministic addresses in freestanding mode.

## Build Flow

From `makefile`:

- Assemble `.s` to `.o`
- Compile `.cpp` with `-m32 -nostdlib -fno-rtti -fno-exceptions`
- Link with `ld -melf_i386 -T linker.ld`
- Build ISO with GRUB

Also notable: `shell_program.s` is assembled into raw binary and linked as blob object (`shell_program_blob.o`).

## Mental Model

Think of boot in 3 layers:

1. **Firmware/bootloader layer**: GRUB loads ELF
2. **Assembly bridge layer**: `loader.s` creates a safe C++ environment
3. **Kernel layer**: `kernelMain` builds all runtime subsystems
