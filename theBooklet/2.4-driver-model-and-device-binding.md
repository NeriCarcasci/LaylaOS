# Chapter 2.4 — Driver Model and PCI Device Binding

You already enumerate PCI. This chapter teaches how discovered hardware turns into active drivers.

## Files to Read

- `driver.h` / `driver.cpp`
- `pci.h` / `pci.cpp`
- `kernel.cpp` (driver manager lifecycle)

## Architecture in This Kernel

### Base abstraction

`Driver` exposes:

- `Activate()`
- `Deactivate()`
- `Reset()`
- `TypeID()`

### Registry

`DriverManager` stores up to 256 pointers and can:

- add drivers
- activate all
- retrieve by index

### Binding point

`PeripheralComponentInterconnect::GetDriver(...)` decides:

- ATA driver for class/subclass `0x01/0x01`
- RTL8139 for network class/subclass with vendor/device `0x10EC/0x8139`

So PCI discovery is separated from driver activation.

## Boot-Time Flow

1. `kernelMain` creates `DriverManager`.
2. PCI scans buses/devices/functions.
3. For each valid descriptor, PCI tries `GetDriver`.
4. Matching drivers are added to manager.
5. `driver_manager.ActivateAll()` initializes hardware.
6. Kernel later searches by `TypeID()` for ATA / NIC wiring.

This is a clean and extensible pattern.

## Why This Matters

This is your first kernel-level form of dependency injection:

- Hardware discovery decides availability.
- Higher layers consume interfaces, not hardcoded globals.

## Practical Next Extension

Add one more driver class:

- Keep `GetDriver` as single selection point.
- Add new `TypeID()` constant.
- Reuse manager activation order.

## Practice Questions

- Why is activation separated from discovery?
- Why is `TypeID()` lookup in `kernel.cpp` useful?
- What would break if you directly constructed ATA/NIC without PCI checks?
