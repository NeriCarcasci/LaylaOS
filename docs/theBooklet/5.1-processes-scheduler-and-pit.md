# Chapter 5.1 — Process Model, PIT, and Scheduler

This chapter covers how your kernel runs Ring 3 code cooperatively through IRQ-driven preemption.

## Files

- `process.h` / `process.cpp`
- `scheduler.h` / `scheduler.cpp`
- `kernel.cpp` (PIT programming + scheduler creation)

## Process Representation

`Process` contains:

- pid/state/exit/parent/wait fields
- kernel stack buffer (4 KiB)
- saved CPU stack pointer (`cpu_state.esp`)
- per-process page directory pointer

Constructor builds an initial interrupt-return stack frame so first schedule can enter Ring 3 at process entry point.

## Ring 3 Entry Setup

In `process.cpp`, stack frame includes:

- user segments (`0x1B` code, `0x23` data)
- user ESP based on pid slot
- `EFLAGS` with IF set
- register/save frames matching ISR stub pop order

This is why returning into user mode works without a dedicated fork/exec loader yet.

## Scheduler Mechanics

`SchedulerTimer` is an interrupt handler on vector 32 (IRQ0).

Every PIT tick:

1. running process marked READY and ESP saved
2. next READY process chosen round-robin
3. page directory switched
4. TSS kernel stack updated
5. next saved ESP returned to ISR path

ISR return restores that context.

## Timing Source

`kernel.cpp` programs PIT channel 0 via ports `0x43` and `0x40` with divisor `11932`.

`pit_ticks` increments each timer interrupt and is reused by syscall timeouts.

## State Transitions You Use

- READY ↔ RUNNING (scheduler)
- RUNNING → BLOCKED (read/wait/network waits)
- RUNNING → DEAD (exit syscall)
- BLOCKED → READY (wake paths)
