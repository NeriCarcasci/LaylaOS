# Chapter 1.3 — TSS and Ring Transitions in Your Kernel

After GDT/IDT, the next critical CPU concept is the **Task State Segment (TSS)**.

In your codebase, TSS is used for one primary purpose: when the CPU transitions from **Ring 3 → Ring 0** (for interrupts/syscalls), it needs a trusted kernel stack.

## Where It Lives in Your Code

- `tss.h` / `tss.cpp`
- `gdt.h` / `gdt.cpp` (TSS descriptor + `ltr`)
- `process.cpp` (per-process kernel stack top)
- `scheduler.cpp` (`tss->SetKernelStack(...)` on context switch)

## Core Idea

The CPU does **not** keep using user stack when privilege changes.

On privilege transition, it loads:

- `SS0` (kernel data segment)
- `ESP0` (kernel stack pointer)

from TSS. In your implementation:

- `TaskStateSegment::TaskStateSegment()` initializes `ss0 = 0x10`
- scheduler updates `esp0` to active process kernel stack top

## What Your GDT Does with TSS

In `GlobalDescriptorTable` constructor:

- Creates TSS descriptor with `access = 0x89` (present, system segment, 32-bit available TSS)
- Loads GDTR via `gdt_flush`
- Executes `ltr` with TSS selector

That means CPU now knows where TSS is and can do safe ring transitions.

## Runtime Flow (Important)

1. Scheduler picks next process.
2. Scheduler switches page directory if needed.
3. Scheduler sets `tss.esp0 = next->KernelStackTop()`.
4. Later, interrupt/syscall from Ring 3 occurs.
5. CPU enters Ring 0 using TSS stack pointer.
6. ISR/syscall runs on kernel stack, not user stack.

Without this, user→kernel transitions will corrupt state or fault.

## Why This Design Is Good for Learning

- Simple: one global TSS object
- Correct enough for privilege transitions
- Makes stack ownership explicit in scheduler

## Common Pitfalls

- Forgetting `ltr` after creating TSS descriptor
- Not updating `esp0` when switching process
- Wrong segment selector constants (`0x10`, `0x1B`, `0x23`)
